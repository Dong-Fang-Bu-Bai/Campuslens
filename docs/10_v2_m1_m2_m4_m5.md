# 第三周 V2：M1、M2、M4、M5 完成记录

日期：2026-06-01

本记录对应细化阶段 2 第三周计划中的 M1、M2、M4、M5。实现范围以当前工程代码、接口契约和测试结果为准。

## M1 图片上传与地标检索主流程

`POST /api/search/upload` 已从运行期临时编号改为数据库记录编号。后端保存上传图片后调用算法服务，按 `landmarkCode` 补齐地标信息，并把上传图片路径、Top-5 结果快照、最高分候选、最高分、检索状态、提示信息和游客身份写入 `search_record` 表。

检索状态当前分为 `success`、`low_confidence`、`empty_result`、`algorithm_unavailable`。算法服务不可用或返回空候选时，接口仍返回明确 `message`、空 `results` 和 `lowConfidence=true`，不伪造演示结果。

管理员轻量登录接口为 `POST /api/admin/auth/login`。第三周仅使用 `admin/admin` 作为本地演示账号，不实现密码加密、权限分级和操作审计。

## M2 地标图像库与元数据

L01-L10 地标元数据继续由 `landmark` 表和 `database/migration/V3__seed_landmarks_and_admin.sql` 统一维护。`GET /api/landmarks/{id}` 返回编号、中文名、英文名、类型、简介、详细介绍、位置说明、地图百分比坐标、代表图路径和样本图列表，能够支撑详情弹窗、详情面板和地图标注。

前端运行时优先读取后端地标数据；若本地代表图文件未实际放入 `frontend/public/images/landmarks/`，页面使用内置示例图作为显示兜底，不影响接口字段和地图坐标验收。

## M4 结果展示与地图导览

前端 Top-5 卡片已展示排名、地标名称、匹配分、匹配等级、马氏距离、简介、详情入口和反馈入口。详情弹窗会按地标 ID 拉取 `GET /api/landmarks/{id}`，并提供跳转到地图标注的入口。

地图导览继续使用 `frontend/public/campus-map.png` 和地标百分比坐标。结果页、详情弹窗、地图标注和反馈入口通过统一的 `selectedId` 与 `searchRecordId` 联动。侧栏新增游客状态、管理员登录入口和后台入口。

管理员后台页面展示最近检索记录和反馈记录。检索记录展示最高候选、状态、游客身份；反馈记录展示检索 ID、预测地标、确认地标、反馈类型、备注和处理状态。

## M5 反馈与记录统计

`POST /api/feedback` 已写入 `feedback` 表，不再使用内存自增编号。提交前会校验 `searchRecordId` 存在，并校验 `predictedLandmarkId` 来自该次检索 Top-5 快照。`wrong` 反馈必须提供 `confirmedLandmarkId`，`correct` 和 `wrong` 必须提供 `predictedLandmarkId`，`uncertain` 允许不指定确认地标。

后台接口 `GET /api/admin/feedback` 返回最近反馈记录，`POST /api/admin/feedback/{id}/status` 支持将状态更新为 `pending`、`accepted` 或 `ignored`。第三周只完成最小处理闭环，反馈采纳后的样本更新、统计图表和审核日志放到后续迭代。

## 数据库迁移与同步

第三周 V2 引入 Flyway。迁移脚本位于 `database/migration/`：

| 脚本 | 内容 |
| --- | --- |
| `V1__baseline_schema.sql` | 创建基础表结构 |
| `V2__v2_record_feedback_admin_fields.sql` | 为既有库补齐 V2 字段 |
| `V3__seed_landmarks_and_admin.sql` | 写入 L01-L10 元数据和管理员演示账号 |

`schema.sql` 和 `seed_landmarks.sql` 保留为手工初始化和文档核对脚本。成员同步表结构时以 Flyway 迁移脚本为准，不把本机 Docker MySQL volume 当作协作交付物。
