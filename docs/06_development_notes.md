## 1. 实训开发总体规范和要求

### 1.1 实训目标

本次软件开发综合实训要求完成一个软件系统的需求、分析设计与实现。实训强调覆盖软件工程完整生命周期，要求团队在实际项目中综合运用需求获取与规约、面向对象分析设计、分层架构、设计原则与设计模式、数据结构与算法、项目管理工具和技术文档。

开发过程需采用“用例驱动、以架构为核心、迭代演化”的方式推进。也就是说，项目不能只停留在页面或单点功能实现上，而要从用例模型出发，逐步形成需求规约、架构设计、用例实现、代码实现、测试与最终交付物。

### 1.2 实验环境要求

任务书要求准备以下基础环境：

| 类别 | 要求 |
| --- | --- |
| IDE 集成开发环境 | IDEA、Eclipse、VS Code、Visual Studio 等 |
| 版本控制工具 | Git 或 SVN |
| UML 建模工具 | StarUML、UMLet 等 |
| 项目管理工具 | Project、Teambition 或其他可替代工具 |

对于 CampusLens 项目，建议第一周同步确定实际使用环境，例如：前端开发工具、后端框架、数据库、图像特征提取方案、向量检索方案、UML 建模工具和协作平台。

### 1.3 开发过程要求

实训周期为 5 周，采用迭代式开发。开发过程划分为初始阶段、细化阶段、构造阶段和交付阶段，其中细化阶段包含 2 次迭代，其余阶段各包含 1 次迭代。每次迭代周期为 1 周，每周都需要有计划、交付版本或交付文档。

| 周次 | 阶段 | 主要要求 | 交付重点 |
| --- | --- | --- | --- |
| 第一周 | 项目初始阶段 | 项目启动、项目计划、愿景范围、可行性分析、风险评估、初始用例模型、核心用例规约、非功能需求、开发环境、界面原型 | 项目开发计划、软件需求规约、核心用例规约、界面原型、个人周报 |
| 第二周 | 细化阶段：第一次迭代 | 制定迭代计划，完善用例模型和需求规约，选取核心业务功能主要场景进行分析设计与实现，完成模块集成与测试 | 架构文档、用例实现规约、迭代版本 V1、个人周报 |
| 第三周 | 细化阶段：第二次迭代 | 继续完善用例模型、需求规约、架构文档和用例实现，选取核心业务功能其他候选场景，建立架构基线 | 迭代版本 V2、架构基线、个人周报 |
| 第四周 | 构造阶段 | 实现其他业务功能用例场景，模块集成与测试，完善用例实现 | 迭代版本 V3、个人周报 |
| 第五周 | 交付阶段 | 系统集成测试、缺陷修复、软件重构、最终版本、答辩 | 最终版本 V4、项目总结报告、答辩 PPT、运行视频、代码与配置说明、个人周报 |

### 1.4 系统实现要求

任务书对系统实现提出了以下总体要求：

1. 软件系统必须完成需求、分析设计与实现，不能只做静态原型或单一演示页面。
2. 需求应基于用例模型和用例规约获取。
3. 设计应采用分层架构。
4. 应基于用例实现产生设计模型。
5. 应运用设计原则和设计模式提升软件质量。
6. 系统需提供后端服务。
7. 可开发 Web 或 APP。
8. 系统应支持可扩展性和可维护性。

### 1.5 分组与角色要求

项目以小组为单位完成，每组 3-6 人，并推举 1 名项目经理。任务书明确要求项目角色按业务功能垂直划分，每位成员的开发任务应涉及不同关注层，例如视图层、业务逻辑层、数据访问层。

因此 CampusLens 不宜按“一个人只做前端、一个人只做后端、一个人只写文档”的方式简单拆分。更合适的方式是按业务用例或业务模块划分责任，例如“图像检索主流程”“地标资料管理”“反馈闭环”“地图与导览展示”等，每位成员都能在自己的模块中覆盖界面、业务、数据或接口中的多个层次。

## 2. 任务三具体要求

### 2.1 任务名称

任务三原题为：校园地标智能检索与导览系统（以图搜图工具）。

本组选定项目名：

| 项目英文名 | 项目中文名 |
| --- | --- |
| CampusLens | 校园慧眼 |

“CampusLens”强调通过照片识别校园地标，“校园慧眼”强调系统具备面向校园环境的智能识别、检索和导览能力。

### 2.2 项目背景

新生或访客对校园环境不熟悉，可能无法仅凭建筑外观判断自己所在位置或目标地点。任务三要求用户拍摄一张校园照片后，系统能够识别照片中的建筑或景观，并返回相关介绍信息，例如历史、开放时间、室内地图等。

系统应服务于学生日常拍照打卡、问路导览、校园开放日和新生入学周等场景。

### 2.3 核心功能要求

任务书列出的核心功能如下：

| 编号 | 功能 | 具体要求 |
| --- | --- | --- |
| TF-01 | 图像库构建 | 采集校园或大学城主要建筑、校门、雕像、湖泊等景观照片；每种地标至少 20 张不同角度和光照样本；存储名称、经纬度、简介等元数据 |
| TF-02 | 图像检索 | 用户上传手机拍摄照片；系统提取深度学习特征；通过余弦相似度匹配 Top-5 地标；返回名称、相关图片和知识库信息 |
| TF-03 | 特征加速 | 使用 FAISS 实现高效向量相似性检索；当前算法服务已按 DINOv2 + FAISS 路线实现框架 |
| TF-04 | 前端交互 | 使用微信小程序或 Web 页面上传图片；展示检索结果及地图位置 |
| TF-05 | 确认反馈 | 用户可纠正检索结果；反馈用于增量学习或调整特征权重 |

### 2.4 对任务三的关键理解

任务三不是单纯的图片上传系统，也不是只做地图展示系统。它的核心是“以图搜图”，即围绕地标图片库、图像特征、相似度检索和地标知识展示形成闭环。

最小可交付版本应至少打通以下链路：

1. 管理端或离线脚本录入校园地标与样本图片。
2. 系统为样本图片生成特征向量。
3. 用户上传待识别图片。
4. 后端提取上传图片特征。
5. 检索模块返回 Top-5 相似地标。
6. 前端展示地标名称、匹配图片、简介、位置和置信度。
7. 用户确认或纠正结果，系统记录反馈。

## 3. CampusLens 产品剖析

### 3.1 产品定位

CampusLens 是面向校园新生、访客和校内用户的校园地标智能检索与导览系统。用户通过上传或拍摄校园照片，系统识别可能的地标并提供位置、介绍和导览信息。

### 3.2 目标用户

| 用户类型 | 典型需求 |
| --- | --- |
| 新生 | 识别陌生建筑，了解用途、开放时间和路线 |
| 访客 | 在开放日、参观或办事时快速确认地点 |
| 在校学生 | 拍照打卡、查找建筑信息、确认路线 |
| 管理员 | 维护地标资料、上传样本图片、处理用户反馈 |
| 项目团队 | 持续扩充图片库、优化检索效果、维护系统稳定性 |

### 3.3 核心业务场景

1. 用户在校园内拍摄某建筑照片，上传后获得 Top-5 候选地标。
2. 用户点击最匹配地标，查看简介、样本图、经纬度和地图位置。
3. 用户发现识别错误，选择正确地标并提交反馈。
4. 管理员新增一个校园地标，填写名称、简介、经纬度并上传样本图。
5. 系统重新生成或更新特征向量索引，提高后续检索效果。

### 3.4 第一版边界

第一版应优先保证“可运行、可演示、可迭代”。建议第一版纳入以下范围：

| 范围 | 第一版建议 |
| --- | --- |
| 地标数量 | 选取 5-10 个代表性地标 |
| 样本图片 | 每个地标先收集并整理 20 张以上样本，满足任务书要求 |
| 检索算法 | 先实现可解释的基础方案：深度特征提取 + 余弦相似度；数据量小时可直接遍历 |
| 前端形式 | 优先 Web 页面，开发与演示成本更低；若小组熟悉小程序，可在后续迭代扩展 |
| 地图能力 | 第一版展示地图点位或跳转地图位置即可，不必一开始实现复杂导航 |
| 反馈能力 | 第一版记录用户纠错信息，后续再用于增量学习或权重调整 |

第一版暂不建议追求过重的能力，例如实时摄像头识别、复杂室内地图、自动爬取全校建筑数据、真正在线训练模型等。这些内容可以作为后续优化项或答辩亮点，但不应压过主线交付。

## 5. 第一周应该做什么

根据任务书，第一周属于“项目初始阶段”，硬性任务不是完成完整系统，而是完成立项、计划、需求、用例、环境和原型。

### 5.1 老师要求的第一周硬性事项

| 序号 | 第一周事项 | CampusLens 对应产出 |
| --- | --- | --- |
| 1 | 项目启动 | 明确项目名 CampusLens / 校园慧眼，确定小组成员、项目经理、角色分工和沟通方式 |
| 2 | 提交项目计划 | 编写项目开发计划，包含 5 周里程碑、WBS、人员分工、工作量估算、进度安排、评审计划 |
| 3 | 定义项目愿景、范围和用户关注 | 明确系统服务对象、核心价值、首版边界、暂不实现内容 |
| 4 | 进行可行性分析 | 分析数据采集、图像特征提取、向量检索、地图展示、Web 或小程序实现的可行性 |
| 5 | 进行风险评估 | 识别图片数据不足、模型效果不稳定、FAISS 接入成本、地图能力、时间不足等风险 |
| 6 | 建立初始用例模型 | 绘制系统用例图，列出普通用户、管理员、检索服务等参与者与核心用例 |
| 7 | 针对核心用例描述用例规约 | 至少完成“上传图片检索地标”“查看地标详情”“提交纠错反馈”等核心用例规约 |

### 5.2 第一周建议交付清单

第一周结束时，建议至少能够拿出以下成果：

## 6. 除老师要求外，第一周建议额外实现的准备工作

为了让第二周能直接进入核心功能迭代，第一周建议额外完成以下工作。这些不一定是老师硬性要求，但会明显降低后续风险。

### 6.1 打通最小技术链路

建议第一周做一个很小的技术验证，不要求功能完整，但要证明关键技术路线可行：

1. 准备 2-3 个地标的少量样本图。
2. 选定一个图像特征提取方案。
3. 能够把图片转换成向量。
4. 能够计算两张图片向量的余弦相似度。
5. 能够返回一个简单的 Top-N 结果。

这个验证可以先用脚本完成，不必一开始接入完整前后端。它的价值是尽早发现模型、依赖、运行环境和数据质量问题。

### 6.2 建立图片采集规范

任务书要求每种地标至少 20 张不同角度和光照样本。第一周应制定统一采集规范，避免后续数据混乱：

| 项目 | 建议规范 |
| --- | --- |
| 地标命名 | 使用统一中文名和英文/拼音标识 |
| 图片数量 | 每个地标不少于 20 张 |
| 拍摄角度 | 正面、侧面、远景、近景、入口、特殊标志物 |
| 光照条件 | 白天、阴天、傍晚等尽量覆盖 |
| 文件命名 | `landmarkCode_angle_light_index.jpg` |
| 元数据 | 地标名称、类型、经纬度、简介、拍摄说明 |
| 质量控制 | 删除严重模糊、遮挡过多、主体不明确的图片 |

### 6.3 先统一接口契约

建议第一周就定义核心接口草案，第二周开发时可以减少前后端扯皮：

| 接口 | 用途 |
| --- | --- |
| `POST /api/search/upload` | 上传图片并返回 Top-5 检索结果 |
| `GET /api/landmarks/{id}` | 获取地标详情 |
| `GET /api/landmarks` | 获取地标列表 |
| `POST /api/feedback` | 提交识别纠错反馈 |
| `POST /api/admin/landmarks` | 新增地标 |
| `POST /api/admin/landmarks/{id}/images` | 上传地标样本图片 |
| `POST /api/admin/index/rebuild` | 重建特征索引 |

算法服务由后端内部调用，统一接口为 `POST /api/v1/search`、`POST /api/v1/index/rebuild`、`GET /api/v1/index/stats` 和 `GET /api/v1/health`。

第一周只需形成接口草案和样例 JSON，不一定全部实现。

### 6.4 初始化工程骨架

建议第一周完成可运行的空工程或最小骨架：

1. 前端项目能启动，包含主要页面路由。
2. 后端项目能启动，包含健康检查接口。
3. 数据库连接方式确定，建表脚本或实体草案形成。
4. 图片上传目录或对象存储方案确定。
5. 检索模块目录预留，后续可接入模型和索引。
6. README 写清楚本地启动方式。

这项工作能确保第二周每位成员可以并行开发，而不是把时间浪费在环境配置上。

### 6.5 制定验收指标

建议第一周提前定义后续演示和答辩可量化指标：

| 指标 | 建议口径 |
| --- | --- |
| 地标覆盖数 | 第一版 5-10 个，最终版可扩展 |
| 样本图片数量 | 每个地标至少 20 张 |
| 检索结果 | 返回 Top-5 候选地标 |
| 响应时间 | 小规模数据下 3 秒内返回 |
| 准确率 | 用测试图片统计 Top-1、Top-5 准确率 |
| 反馈闭环 | 至少能记录用户纠错并在管理端查看 |
| 可维护性 | 新增地标后能重新生成或更新索引 |

## 9. 主要风险与应对

| 风险 | 影响 | 应对策略 |
| --- | --- | --- |
| 样本图片数量不足或质量不稳定 | 检索准确率低，无法体现以图搜图效果 | 第一周就启动采集，制定拍摄规范，每个地标至少 20 张 |
| 深度学习环境配置耗时 | 第二周核心功能受阻 | 第一周先做最小脚本验证，当前路线采用本地 DINOv2 模型文件，需提前准备模型和依赖 |
| FAISS 索引维护成本较高 | 影响主流程进度 | 当前框架已接入 FAISS，后续重点验证索引重建、样本质量和部署环境 |
| 地图能力实现复杂 | 影响前端展示进度 | 第一版先展示经纬度和地图链接，后续再增强 |
| 分工过于横向 | 成员成果难以覆盖不同关注层 | 按用例或业务模块垂直分工，每个模块覆盖页面、接口和数据 |
| 文档滞后于代码 | 阶段交付评分受影响 | 每周按模板同步更新计划、需求、架构、用例实现和周报 |
| 上传图片存在安全风险 | 后端稳定性和安全性下降 | 限制文件类型、大小和存储路径，避免直接执行或公开任意文件 |

## 10. 结论

第一周最重要的目标是完成 CampusLens 的“立项与可迭代开发基础”，而不是仓促堆功能。必须完成的内容包括项目计划、需求规约、核心用例规约、非功能需求、风险评估、开发环境和界面原型。

在此基础上，建议额外完成最小技术链路验证、图片采集规范、接口契约草案、工程骨架和验收指标。这样第二周进入细化阶段第一次迭代时，团队可以直接围绕“上传图片并返回 Top-5 地标”这一核心主流程进行分析设计、实现、集成和测试。
