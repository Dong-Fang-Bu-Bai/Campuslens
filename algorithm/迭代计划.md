1. 迭代目标
本周处于软件开发综合实训的初始阶段，目标不是完成完整业务系统，而是完成项目启动、需求边界、初始用例、技术路线、数据规范、接口契约、仓库骨架和后续迭代准备。本迭代计划用于明确五名成员在第一周应完成的任务，并为第二周进入“上传图片并返回 Top-5 地标”的最小闭环开发提供统一依据。
•	明确 CampusLens 的项目愿景、目标用户、功能范围和不纳入首版的边界。
•	完成项目开发计划、软件需求规约、核心用例规约、界面原型和个人周报等第一周交付物。
•	建立前端、后端、算法、数据库、数据集和文档目录的协作口径。
•	完成首批 10 个校园地标的清单、图片采集规范和数据库初稿。
•	结合最新 dev 分支中已形成的 DINOv2 + FAISS 算法服务，继续补齐前后端工程骨架和联调依据。
2. 当前 dev 分支基础
截至 2026-05-20，dev 分支已经形成以下基础，可作为本周任务推进和第二周迭代开发的前置成果：
•	仓库已建立 frontend、backend、algorithm、database、datasets、api、docs 等技术目录。
•	docs 目录已包含项目总览、Git 协作规则、接口契约说明、数据字典、图片采集规范、模块归属与协作边界和开发文档。
•	api 目录已包含 OpenAPI 草案和 REST Client 接口测试样例，用户端核心接口以 POST /api/search/upload、GET /api/landmarks、GET /api/landmarks/{id} 和 POST /api/feedback 为主。
•	database 目录已包含 schema.sql 和 seed_landmarks.sql，覆盖 landmark、landmark_image、image_feature、search_record、feedback、admin_user 等核心数据表。
•	datasets/landmarks 已按 L01-L10 建立首批地标目录，并保留 README 和 .gitkeep。
•	algorithm 目录已形成 Python FastAPI 算法服务框架，包含 DINOv2 特征提取、马氏距离统计检索、统计参数重建接口、健康检查、模型验证脚本、CPU/GPU 安装脚本和 Docker 配置。
•	frontend 和 backend 目前仍以 README 和职责说明为主，尚未形成可启动的 Vue 与 Spring Boot 工程。

4. 五人任务分配
本项目按垂直业务模块进行分工。每位成员在第一周应围绕自己负责的业务模块完成需求、设计、接口、数据、文档和后续实现准备，避免只承担单一技术层面的任务。
4.1 马启凡：项目经理 / M1 图片上传与地标检索主流程
•	牵头确认项目名称、项目愿景、目标用户、首版范围和第一周提交口径。
•	整理并维护项目开发计划，明确五周里程碑、WBS、工作量估算、风险评估和评审安排。
•	维护 README、Git 分支协作规则和接口联调口径，确保 main、dev、feature 分支职责清晰。
•	负责 M1 主流程的需求拆分，明确上传图片、文件校验、保存上传图、创建检索记录、调用算法服务和组装 Top-5 返回结果的职责边界。
•	参与制定 POST /api/search/upload 的请求、响应、错误码和异常处理规则，并与 M3 算法服务接口保持一致。
•	协调 M2 提供基础地标与样本图片，协调 M3 提供算法接口，协调 M4 与 M5 预留结果展示和反馈入口。
•	准备第一周个人周报，并汇总小组文档归档结构。
4.2 叶炳良：M2 地标图像库与元数据管理
•	负责首批 10 个地标对象的清单整理，统一地标编号、中文名称、英文或拼音名称、类型、位置描述和简介口径。
•	制定并维护图片采集规范，包括每个地标不少于 20 张、拍摄角度、光照条件、质量筛选、代表图选择和命名规则。
•	维护 datasets/landmarks 下 L01-L10 目录结构，保留 README 和 .gitkeep，确保数据集目录可追踪但不直接提交大量原图。
•	完善 landmark、landmark_image 相关数据字段说明，并与 database/schema.sql、seed_landmarks.sql 和 docs/03_data_dictionary.md 保持一致。
•	规划地标资料维护方式，明确第二周可通过数据库脚本、后端接口或简化后台维护地标元数据。
•	参与确定图片与向量文件存储方式，明确 MySQL 只保存路径和元数据，不直接保存大文件。
•	准备 M2 模块第一周周报，记录地标清单、采集规范和数据准备进展。
4.3 周子栋：M3 图像特征提取与向量索引服务
•	负责确认图像检索技术路线，当前 dev 分支采用 DINOv2 ViT-B/14 提取图像特征，使用地标均值、正则化协方差和协方差逆矩阵完成马氏距离统计检索。
•	维护 algorithm FastAPI 服务，包括健康检查、地标检索、统计参数重建、统计参数状态等接口。
•	整理 DINOv2 模型文件准备方式、模型存放路径、CPU/GPU 安装方式、依赖安装和启动步骤。
•	使用 verify_model.py、check_service.py 和接口样例验证算法服务能在本地环境中完成启动、模型检查、统计参数构建和检索调用。
•	明确样本图片输入目录、特征缓存目录、统计参数目录和返回结果字段，支撑 M1 后端主流程集成。
•	补充算法验证记录，说明模型选择原因、特征维度、统计参数重建流程、低置信度处理和后续性能测试计划。
•	准备 M3 模块第一周周报，记录算法服务框架、依赖风险、模型文件准备和后续联调需求。
4.4 洪传凯：M4 检索结果展示与地图导览
•	负责用户端页面原型设计，至少覆盖图片上传页、Top-5 检索结果页、地标详情页、反馈入口和静态地图展示区域。

•	整理地标详情页展示内容，包括地标简介、样本图、位置描述、开放信息、地图标注和返回结果页的跳转关系。
•	与 M1 确认结果页接收的数据结构，与 M2 确认代表图片和地标详情字段，与 M5 确认反馈入口位置。
•	为第二周前端 Vue 工程落地准备页面结构、路由草案和主要组件划分。
•	准备 M4 模块第一周周报，记录原型产出、地图素材使用和前端实现计划。
4.5 庄子杰：M5 用户反馈纠错与检索记录统计

6. 本周收尾要求
1.	所有文档应使用正式实验报告语气，避免只写讨论过程或聊天式说明。
2.	涉及接口字段、数据库字段、数据集目录和算法返回格式的变更，应同步更新 docs、api、database 和 README 中的对应说明。
3.	数据集原图、模型权重、运行日志和本地缓存不直接提交到 Git；统计参数与索引文件是否提交由 M3 后续统一确认，当前不在本次文档同步中扩大处理。
4.	第一周结束前应至少完成核心用例规约、页面原型、个人周报和算法验证记录的归档，否则初始阶段交付物仍不完整。
5.	第二周进入细化阶段第一次迭代时，应优先围绕“上传图片 -> 调用算法服务 -> 返回 Top-5 地标 -> 页面展示结果”打通最小闭环。
7. 与后续迭代的衔接
第一周输出的计划、需求、接口、数据、原型和算法验证材料，是第二周 V1 主流程实现的输入。第二周应以 M1、M2、M3 为主线，由 M1 打通后端上传和检索记录，由 M2 提供基础地标与样本图片，由 M3 提供可调用的算法服务。M4 在此基础上完成结果页展示，M5 预留反馈入口和记录字段，确保第三周能够继续完善详情、地图标注、反馈闭环和统计参数机制。

根据初始阶段迭代计划，本轮已补齐 M1、M2、M4、M5 的工程与文档准备。后端最小工程、前端原型、地标元数据、反馈接口和原型截图均已归档到 Campuslens 仓库。
M1 完成后端最小接口和上传检索演示流程；M2 完成首批地标元数据补充；M4 完成 Vue 前端原型和静态地图标注；M5 完成反馈接口、反馈表单和统计口径说明。
